# Norbert
Norbert is an open-source runtime that gives local models the tools, memory, and feedback loops they need to do real software engineering.


# The Norbert Manifesto

## Local Models. Real Tools. Tight Loops.

Norbert is an open-source development agent built on a simple belief:

**The future of AI-assisted software development will be local.**

The next major leap in developer productivity will not come only from larger models, longer prompts, or more expensive cloud APIs.

It will come from giving capable local models the right environment:

- Useful tools
- Durable context
- Project memory
- Clear feedback
- Safe execution
- Repeatable loops

Models do not always need to be smarter.

They need to be able to work.

---

## Why Norbert?

Norbert is named in honor of **Norbert Wiener**, whose work on cybernetics focused on feedback, control, communication, and systems that improve their behavior by responding to results.

That idea is at the center of this project.

A development agent should not make one guess and stop.

It should:

1. Observe the project.
2. Form a plan.
3. Take an action.
4. Measure the result.
5. Correct its approach.
6. Repeat until the work is complete.

The name Norbert also carries the meaning **“bright north”** or **“northern light.”**

That meaning reflects what the project should be:

A guide through complex systems.

A source of direction without removing human control.

A local intelligence that helps developers see the next step.

---

## The Future of Development Is Local

Developer AI should run locally whenever possible.

Local-first development means:

- Your source code stays on your machine.
- Your secrets stay private.
- Your tools continue to work offline.
- Your workflow is not dependent on a single vendor.
- Your development environment remains under your control.
- Your agent can be inspected, modified, extended, and trusted.

Cloud models may still be useful.

They should be an option, not a dependency.

**Local should be the default. Remote should be the escape hatch.**

---

## The Model Is Not the Product

A language model by itself is not a development system.

It may be able to explain code, suggest a function, or produce a plausible answer. But real software development requires more than text generation.

It requires interaction with reality.

A useful development agent needs:

- Access to the codebase
- Search
- File editing
- Build tools
- Test execution
- Compiler feedback
- Documentation
- Version control
- Project history
- Domain knowledge
- Runtime inspection
- Memory
- Permission boundaries

The model is one component.

The product is the system around it.

**The loop is the product.**

---

## Loops Over Raw Intelligence

We reject the assumption that progress depends only on making models larger or more intelligent.

A capable model inside a poor workflow will still fail.

A smaller model inside a strong loop can accomplish meaningful work.

A Norbert development loop should look like this:

1. Read the relevant code.
2. Understand the task.
3. Inspect project rules and documentation.
4. Form a plan.
5. Make a focused change.
6. Build the project.
7. Read the errors.
8. Correct the implementation.
9. Run the tests.
10. Review the resulting diff.
11. Explain what changed.
12. Repeat when necessary.

This is not a workaround for limited intelligence.

This is software engineering.

Developers do not solve every problem in one attempt. They inspect, test, learn, and revise.

AI development systems should work the same way.

---

## Tools Are the Real Multiplier

Models become useful when they can use tools reliably.

Norbert should be able to:

- Search a codebase
- Read and edit files
- Run commands
- Build applications
- Execute tests
- Inspect compiler and runtime errors
- Query local documentation
- Use project-specific knowledge
- Interact with version control
- Call MCP servers
- Track tasks and decisions
- Verify its own work
- Stop when human approval is required

Better prompting can improve an answer.

Better tools can transform what the agent is capable of doing.

Norbert should prefer grounded action over confident guessing.

---

## Context Is a First-Class Feature

A developer agent must understand the project in which it is working.

Not only the active file.

Not only a pasted snippet.

Not only the most recent chat message.

Useful context includes:

- Architecture decisions
- API documentation
- Coding standards
- Build instructions
- Test conventions
- Domain rules
- Known limitations
- Previous attempts
- Recent changes
- Local environment details
- Developer preferences
- Security boundaries

This context should be structured, searchable, durable, and controlled by the developer.

Norbert should not pretend to know what it can inspect.

It should retrieve the information it needs.

---

## Memory Should Serve the Project

Memory should not be an uncontrolled transcript of everything that happened.

It should be deliberate and useful.

Norbert should remember:

- Important architectural decisions
- Project conventions
- Failed approaches and why they failed
- Commands required to build and test
- Developer-approved preferences
- Important relationships between files and systems
- Open questions and unfinished work

Memory should reduce repeated mistakes.

It should not create hidden behavior.

Developers should be able to inspect, edit, export, and delete it.

---

## Open Source Is a Requirement

A system that can read code, execute tools, modify files, and influence what ships must be inspectable.

Developers should be able to understand:

- What Norbert is doing
- What information it is using
- What tools it can access
- What commands it executed
- Why it made a change
- What data leaves the machine
- Where memory is stored

Open source is not only a licensing preference.

It is part of the trust model.

Norbert is for developers who want control, not dependency.

For teams who want transparency, not magic.

For builders who want systems they can own.

---

## The Developer Remains in Charge

Norbert is not built to replace developers.

It is built to increase their leverage.

The human sets the direction.

The human owns the judgment.

The human decides what ships.

Norbert should handle more of the repetitive loop work:

- Searching
- Editing
- Building
- Testing
- Checking
- Documenting
- Comparing
- Retrying

The developer remains the architect, reviewer, and final authority.

Automation should expand human capability without removing human control.

---

## Small Models Can Do Big Work

A local model does not need to know everything in advance.

It needs the ability to look things up.

It does not need to predict the perfect answer on the first attempt.

It needs the ability to test its answer.

It does not need to solve an entire project in one response.

It needs the ability to decompose the work and iterate.

A well-designed loop can compensate for a great deal of missing raw intelligence.

A powerful system can emerge from:

- A capable local model
- Good retrieval
- Focused tools
- Strong feedback
- Persistent memory
- Careful orchestration
- Human oversight

Norbert should prove that useful local development agents do not require the largest available model.

---

## Model Agnostic by Design

Norbert should not belong to one model provider.

Developers should be free to choose models based on:

- Their hardware
- Their privacy requirements
- Their programming languages
- Their context needs
- Their speed requirements
- Their licensing preferences
- Their budget
- The task being performed

Different models may be best for planning, coding, reviewing, searching, or summarizing.

Norbert should provide the loop and tool environment without locking the developer into a single intelligence provider.

Models will change.

The developer’s workflow should survive those changes.

---

## Verification Is Part of Intelligence

Generated code is not complete because it looks correct.

It is complete when it has been checked against reality.

Norbert should use:

- Builds
- Tests
- Linters
- Type checkers
- Static analysis
- Runtime checks
- Diff review
- Documentation validation
- Security scanning
- User-defined acceptance criteria

Verification tools are not separate from the agent’s intelligence.

They are part of it.

A model that can test and correct its work is more useful than a model that can only produce convincing text.

---

## Logs Over Magic

Norbert should make its work understandable.

A developer should be able to see:

- The plan
- The files inspected
- The tools used
- The commands executed
- The changes made
- The tests run
- The failures encountered
- The corrections attempted
- The final result
- Any remaining uncertainty

Hidden behavior damages trust.

Clear records create accountability.

The goal is not to expose endless internal reasoning.

The goal is to provide enough evidence for a developer to review the work confidently.

---

## Built for Real Development

Norbert should be measured against real projects, not staged demonstrations.

It should work with:

- Large codebases
- Incomplete documentation
- Legacy systems
- Multiple programming languages
- Custom build environments
- Local APIs
- Proprietary domain knowledge
- Long-running tasks
- Failed builds
- Imperfect models
- Real developer constraints

We value:

- Reliability over spectacle
- Repeatability over cleverness
- Evidence over confidence
- Testable changes over plausible guesses
- Local control over vendor lock-in
- Simple tools over unnecessary abstraction
- Practical workflows over hype

If Norbert cannot improve a real developer’s day, it is not good enough.

---

## Core Principles

### 1. Local by Default

The system should run on the developer’s machine whenever practical.

### 2. The Loop Is the Product

Norbert exists to coordinate observation, action, feedback, correction, and verification.

### 3. Tools Before Tricks

Reliable tool access is more valuable than increasingly elaborate prompting.

### 4. Context Over Guesswork

Norbert should ground its work in project files, documentation, tests, memory, and runtime feedback.

### 5. Human Authority

The developer must remain in control of permissions, changes, and what ultimately ships.

### 6. Open and Extensible

Developers should be able to inspect, modify, replace, and extend every major part of the system.

### 7. Model Agnostic

Norbert should support multiple local and optional remote models without depending on one provider.

### 8. Privacy Is a Feature

Source code, credentials, business logic, and project history should not leave the machine by default.

### 9. Logs Over Magic

Norbert should provide clear evidence of what it did and what happened.

### 10. Verification Matters

Builds, tests, linters, analyzers, and runtime checks are essential parts of the development loop.

### 11. Memory Must Be Controllable

Project memory should be inspectable, editable, exportable, and removable.

### 12. Failures Are Feedback

Errors should feed the next iteration, not end the process or be hidden from the developer.

### 13. Composability Over Monoliths

Small tools with clear contracts should be combined into powerful workflows.

### 14. Permission Before Power

The more powerful a tool is, the clearer its boundaries and approval model must be.

### 15. Practicality Over Hype

Every feature should help developers understand, build, verify, or maintain real software.

## What We Are Building

Norbert is an open-source, local-first development agent.

It is not merely a chatbot beside the editor.

It is a loop-driven engineering system that connects:

- Local language models
- Developer tools
- Project documentation
- Search and retrieval
- Build and test systems
- Memory
- MCP servers
- Human review

Norbert should be able to read, reason, act, test, learn from results, and improve its work.

It should treat the developer’s local machine as the center of the development experience.

It should prove that the future of AI-assisted programming does not have to be closed, remote, expensive, or dependent on ever-larger models.

---

## The Mission

Make local AI development powerful, private, transparent, extensible, and practical.

Give developers ownership of their AI workflows.

Build an agent that strengthens human judgment instead of hiding it behind automation.

Prove that the next generation of coding systems will be built not only with smarter models, but with better tools, better context, better feedback, and better loops.

**The future of development is local.**

**The model is not the product.**

**The loop is the product.**
